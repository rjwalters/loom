# v1.0.0 Migration Guide — shepherd/daemon deprecation

> **Audience**: anyone with an installed copy of Loom (via `install.sh` /
> `scripts/install-loom.sh`) who has been running the Python `loom-daemon`,
> the Python `loom-shepherd` CLI, or the `/shepherd` Claude Code slash command.
>
> **Status**: this guide is authoritative for the v1.0.0 release. It collects
> in one place every breaking change introduced by epic #3372 — the deletion
> of the shepherd brain, the daemon brain, and the `/shepherd` skill.

## TL;DR

The Python `loom-daemon` brain (`loom_tools/daemon_v2/`), the shepherd brain
(`loom_tools/shepherd/`), and the `/shepherd` slash command have all been
**removed in v1.0.0**. They were soft-deprecated in 0.9.x (#3376, Phase 2b)
and replaced by:

- **`/loom:sweep <issue>`** — single-issue lifecycle (Curator → Builder →
  Judge → Doctor → Merge). Same phases as `/shepherd`, but maintained.
- **`./.loom/scripts/spawn-loop.sh`** — multi-issue batch driver. Replaces
  the daemon brain for issue work.
- **GitHub Actions cron workflows under `.github/workflows/loom-*.yml`** —
  replaces the support-role daemons (Champion, Curator, Judge, Auditor,
  Guide). Opt-in per workflow.

If you were running `./.loom/scripts/daemon.sh start`, the post-migration
equivalent is:

```bash
LOOM_USE_SPAWN_LOOP=1 ./.loom/scripts/spawn-loop.sh start
# + enable the GitHub Actions workflows you want under .github/workflows/loom-*.yml
```

If you were running `/shepherd <issue>` interactively, the equivalent is:

```text
/loom:sweep <issue>
```

If you were running `claude -p "/shepherd 123"` from a script, the equivalent is:

```bash
claude -p "/loom:sweep 123" --dangerously-skip-permissions
```

If you have downstream code that imports from `loom_tools.daemon_v2` or
`loom_tools.shepherd`, or that reads `.loom/daemon-state.json` or
`.loom/progress/`, those imports will break — see
[Per-CLI breaking changes](#per-cli-breaking-changes) and
[State-file consumers](#state-file-consumers).

## Phase rollout recap

Epic #3372 was rolled out in phases so downstream consumers had time to
migrate. v1.0.0 closes the migration window.

| Phase | Issue | What shipped | When |
|-------|-------|-------------|------|
| Phase 1 | #3374 | Minimal multi-account spawn loop (`./.loom/scripts/spawn-loop.sh`) | 0.9.x |
| Phase 2a | #3375 | GitHub Actions workflows for support roles | 0.9.x |
| Phase 2b | #3376 | Soft-deprecation warnings on every deprecated entry point | 0.9.x |
| Phase 2c | #3377 | Per-consumer disposition inventory (`docs/migration/daemon-state-consumers.md`) | 0.9.x |
| Phase 2d | #3381 | Architect/Hermit cadence design (work-generation triggers) | 0.9.x |
| **Phase 3** | #3378 (meta) | **Hard deletion** of shepherd brain, daemon brain, `/shepherd` | **v1.0.0** |
| Phase 4 | #3382 | Sphere downstream coordination | parallel with v1.0.0 |

This guide is the v1.0.0 (Phase 3) artifact.

## What was deleted

The following code, slash commands, state files, and entry points were
removed in v1.0.0. If you have not migrated, your `install-loom.sh` run will
silently stop installing the files marked with [installer-managed].

### Python packages

- `loom-tools/src/loom_tools/daemon_v2/` — the daemon brain (work generation,
  shepherd scaling, support-role triggers, state-file write loop).
- `loom-tools/src/loom_tools/shepherd/` — the shepherd brain (per-issue
  Curator → Builder → Judge → Doctor → Merge orchestration).
- `loom-tools/src/loom_tools/daemon.py` — the legacy single-file daemon.
- `loom-tools/src/loom_tools/snapshot.py` — the pipeline-health composer.
  The forge-query half was extracted to `forge_snapshot.py` first; only the
  daemon-state-coupled half was deleted.
- `loom-tools/src/loom_tools/milestones.py` — the milestone-report producer
  (heartbeat writer for `.loom/progress/shepherd-*.json`).
- `loom-tools/src/loom_tools/validate_state.py` — validator for a file that
  no longer exists.
- `loom-tools/src/loom_tools/models/daemon_state.py`,
  `loom-tools/src/loom_tools/models/progress.py` — pydantic models for the
  retired files.
- `loom-tools/src/loom_tools/test_failure_analysis.py` — read `.loom/progress/`
  for pattern analysis; retired with the producer.
- `loom-tools/src/loom_tools/common/state.py::read_daemon_state` and
  `::read_progress_files` and `::find_progress_for_issue` — helper APIs for
  the retired files.
- `loom-tools/src/loom_tools/common/systematic_failure.py` — daemon-loop
  failure counter.
- `loom-tools/src/loom_tools/common/deprecation.py` — the Phase 2b warning
  emitter; no longer needed once the deprecation window closes.

### Entry points (`loom-tools/pyproject.toml`)

- `loom-daemon` (Python) — the daemon-brain CLI.
- `loom-shepherd` — the shepherd-brain CLI.
- `loom-validate-state` — validator for `.loom/daemon-state.json`.

The corresponding shell wrappers under `defaults/scripts/` (e.g.
`loom-shepherd.sh`, `daemon.sh`, `daemon-cleanup.sh`,
`rotate-daemon-state.sh`, etc.) were removed in the same wave. See
`docs/migration/daemon-state-consumers.md` §"Shell scripts to retire" for
the authoritative 22-script list.

### Slash commands / agents [installer-managed]

- `defaults/.claude/commands/loom/shepherd.md` — the `/shepherd` skill.
- `defaults/.claude/commands/loom/shepherd-lifecycle.md` — lifecycle
  documentation referenced by `/shepherd`.
- `defaults/.claude/agents/loom-daemon.md` — the daemon-brain subagent.
- `defaults/.claude/agents/loom-shepherd.md` — the shepherd-brain subagent.
- `defaults/roles/loom.md` and `defaults/roles/shepherd.md` — role
  definitions referenced by the deleted subagents.

After your next `install-loom.sh` run against v1.0.0, these files will be
absent from your repository's `.claude/agents/` and `.claude/commands/loom/`.
If you have custom slash commands or hooks that referenced them, those
references will silently no-op.

### State files (gitignored, on disk only)

- `.loom/daemon-state.json` and `.loom/[0-9][0-9]-daemon-state.json`
  archives.
- `.loom/progress/` directory and all `shepherd-*.json` files within.
- `.loom/alerts.json` — daemon alert log.
- `.loom/health-metrics.json` — health-monitor metric series.
- `.loom/stuck-history.json` — stuck-detection intervention log.
- `.loom/daemon-metrics.json` — per-iteration counters.
- `.loom/baseline-health.json` — health-score baseline.

The Rust daemon's `post_init.rs` `.gitignore` writer drops these patterns
too (Phase 3.5 / PR — separate from this docs sweep), so a fresh install
will not regenerate them. Existing on-disk copies are harmless and can be
deleted at your leisure.

### Files explicitly NOT deleted

These look related but are intentionally preserved. Do not delete them
manually.

- `.loom/issue-failures.json` — durable cross-session failure log, consumed
  by the ported `loom-backlog`. Stays.
- `.loom/spawn-loop-state.json` — Phase 1 state file. Stays (this is the
  replacement for `.loom/daemon-state.json`).
- `.loom/sweep-checkpoint/issue-<N>.json` — Phase 1 checkpoint files (#3373).
  Stays.
- `.loom/tokens/` and everything under it — multi-account token rotation.
  Stays.
- Rust `loom-daemon/` and Rust-daemon control scripts
  (`defaults/scripts/start-daemon.sh`, `stop-daemon.sh`,
  `daemon-headless.sh`) — different binary, different concern. Stays.
- `mcp-loom/` — MCP server. Stays.
- `activity.db` — agent activity metrics database. Stays.

## How to enable the replacements

### Spawn-loop mode (replaces `./.loom/scripts/daemon.sh start`)

The spawn loop is **opt-in via env var** to protect existing daemon users
from accidental dual-orchestration. To run a multi-issue batch:

```bash
# One-time opt-in gate (required in 0.9.x; remains required in v1.0.0)
export LOOM_USE_SPAWN_LOOP=1

# Start the loop. It detaches as a background process.
./.loom/scripts/spawn-loop.sh start

# Status / stop
./.loom/scripts/spawn-loop.sh status
./.loom/scripts/spawn-loop.sh stop      # or: touch .loom/stop-spawn-loop
```

State is written to `.loom/spawn-loop-state.json`. Logs go to
`.loom/logs/spawn-loop.log` (loop-level) and
`.loom/logs/sweep-issue-<N>.log` (per-child). Atomic claim locks live under
`.loom/locks/issue-<N>/`.

Tunables (env):

| Variable | Default | Description |
|----------|---------|-------------|
| `MAX_PARALLEL` | 3 | Maximum concurrent `/loom:sweep` children |
| `POLL_INTERVAL` | 30 | Seconds between `loom:issue` polls |
| `SHUTDOWN_GRACE_SEC` | 300 | Seconds to wait for in-flight children at shutdown |
| `LOOM_REPO` | (auto) | Override remote auto-detection (`owner/repo`) |

The spawn loop **does not** generate work (no Architect / Hermit triggers),
**does not** maintain a `shepherd-N` pool, and **does not** drive support
roles. Each ready issue runs its own self-contained `claude -p
"/loom:sweep N"` child. If you need work generation, see the Phase 2d
follow-up (#3381).

### Scheduled support roles (replaces always-running Champion / Curator / Judge / Auditor / Guide)

GitHub Actions cron workflows under `.github/workflows/loom-*.yml` run the
periodic support roles on schedules that match the daemon's historical
intervals.

| Workflow | Role | Schedule (commented in-template) |
|----------|------|----------------------------------|
| `loom-champion.yml` | `/champion` | `*/10 * * * *` |
| `loom-curator.yml`  | `/curator`  | `*/5 * * * *`  |
| `loom-judge.yml`    | `/judge`    | `*/5 * * * *`  |
| `loom-auditor.yml`  | `/auditor`  | `*/10 * * * *` |
| `loom-guide.yml`    | `/guide`    | `*/15 * * * *` |

**Workflows are disabled by default** — each shipped workflow has its
`schedule:` block commented out. To opt in on a fork:

1. Add a `CLAUDE_API_KEY` repository secret (`Settings → Secrets and
   variables → Actions`). Scheduled support roles use one key; multi-account
   token rotation is for per-task spawns only.
2. Uncomment the `schedule:` / `- cron:` lines in each
   `.github/workflows/loom-*.yml` you want to enable.
3. Smoke-test via `workflow_dispatch` (the Actions UI's "Run workflow"
   button) before the next scheduled tick.

Architect and Hermit cadence (work-generation triggers) is **out of scope**
for v1.0.0 — tracked under follow-up #3381.

### `/loom:sweep` for single-issue work (replaces `/shepherd`)

`/loom:sweep <issue>` runs the same five-phase lifecycle as the old
`/shepherd` did:

```text
Curator → Builder → Judge → Doctor (if needed) → Merge
```

Invocation patterns:

```text
/loom:sweep 123                      # Run the full lifecycle for issue 123
/loom:sweep --prs 456 789            # Mode C (#3384): PR-set mode, runs
                                     # Judge / Doctor → Judge / Merge against
                                     # an existing open-PR set without
                                     # re-running Curator or Builder.
```

From a script:

```bash
claude -p "/loom:sweep 123" --dangerously-skip-permissions
```

Checkpoints from #3373 are preserved across the rename — if a sweep child
crashes mid-flight and a `.loom/sweep-checkpoint/issue-<N>.json` exists, the
next run of the sweep skill resumes from the recorded phase.

## Per-CLI breaking changes

The nine operator CLIs that historically consumed `.loom/daemon-state.json`
or `.loom/progress/` were ported as part of Phase 3.1.x. Each has its own
PR and follow-up issue. Below is the field-level summary — for the
authoritative pre/post shape comparison, read the CLI's `--help` and the
linked PR.

### `loom-status` (#3399)

- **Before**: read `.loom/daemon-state.json` for shepherd state, pipeline
  state, warnings; cross-referenced with forge queries.
- **After**: reads `.loom/spawn-loop-state.json` for running tasks (just
  `{issue, pid, started_at, token}`), forge queries for everything else.
- **Dropped fields**: `shepherds.shepherd-N.*` (pool slots gone),
  `pipeline_state.*` (forge is source of truth), `warnings[]`,
  `total_prs_merged`, `last_architect_trigger`, `last_hermit_trigger`.
- **Renamed fields**: `shepherds[].issue` → `tasks[].issue`,
  `shepherds[].task_id` → `tasks[].pid`.
- **Migration**: re-script anything that consumed those dropped fields
  against `gh issue list --json` and `gh pr list --json` directly.

### `loom-backlog` (#3400)

- **Before**: read `.loom/daemon-state.json` for shepherd warnings about
  blocked PRs; cross-referenced with `.loom/issue-failures.json` for
  durable failure history.
- **After**: reads `.loom/issue-failures.json` only. The `daemon-state.json`
  cross-reference path is gone.
- **Dropped fields**: any `warnings[].context.requires_role` heuristics.
- **Migration**: most callers were already preferring `issue-failures.json`;
  this PR removes a fallback path that few callers exercised.

### `loom-stuck-detection` (#3401)

- **Before**: read `.loom/daemon-state.json` shepherd heartbeat fields and
  `.loom/progress/shepherd-*.json` milestone files.
- **After**: reads `.loom/spawn-loop-state.json` task pids (probing via
  `kill -0 <pid>`) and `.loom/sweep-checkpoint/issue-<N>.json` timestamps.
- **Dropped indicators**: `missing_milestone:worktree_created`,
  `extended_work` (no longer tractable without `last_phase` in state).
- **Retained indicator**: `stale_heartbeat` (now derived from checkpoint
  mtime, not progress-file mtime).
- **Migration**: thresholds are env-configurable as before; if you
  scripted against the JSON output, the `stale_heartbeat` block schema is
  unchanged, but the `extended_work` block is gone.

### `loom-completions` (#3402)

- **Before**: read shepherd `output_file` paths from daemon-state.json to
  produce shell completion data.
- **After**: reads `.loom/logs/sweep-issue-*.log` directly.
- **Breaking**: completion shape unchanged. If you wrote a completion
  cache that keyed on `output_file`, re-key on the log path.

### `loom-agent-metrics` (#3404)

- **Before**: daemon-state.json fallback for "is agent currently working".
- **After**: `activity.db` only. The fallback is gone.
- **Migration**: ensure `activity.db` is being populated by your spawn-loop
  / sweep runs (this is unchanged — sweep children write activity entries
  the same way shepherd children did).

### `loom-orphan-recovery` (#3414)

- **Before**: cross-referenced daemon-state.json's `shepherds[]` against
  the forge label set to find issues stuck in `loom:building` with no
  active shepherd.
- **After**: cross-references `.loom/spawn-loop-state.json` tasks against
  the forge.
- **Migration**: invocation unchanged (`--recover`, `--dry-run`, `--json`).
  Output schema unchanged.

### `loom-cleanup` (#3412, renamed from `loom-daemon-cleanup`)

- **Renamed**: `loom-daemon-cleanup` → `loom-cleanup`. The old name is gone.
- **Before**: cleaned up `.loom/progress/shepherd-*.json` files, rotated
  `daemon-state.json` archives, emitted `session-rotation` events.
- **After**: cleans up `.loom/logs/sweep-issue-*.log` (configurable
  retention), purges stale `.loom/locks/issue-<N>/` dirs, no session
  rotation.
- **Migration**: update any cron entry or hook that referenced
  `loom-daemon-cleanup` by name. The new name is `loom-cleanup`.

### `loom-health-monitor` (#3413)

- **Before**: computed a composite health score from daemon-state.json,
  `.loom/health-metrics.json`, `.loom/alerts.json`.
- **After**: reads forge state only (queue depth, PR backlog) and writes a
  simplified `.loom/health-metrics.json` (smaller schema).
- **Dropped fields**: `stuck_agents`, `resource_availability` (these
  required state that no longer exists).
- **Migration**: re-tune your alerting thresholds. The new health score is
  not directly comparable to the old one.

### `loom-clean` (#3415)

- **Before**: read `daemon-state.json` to decide whether a worktree was in
  use by an active shepherd.
- **After**: reads `.loom/spawn-loop-state.json` (`tasks[].issue`) and
  `.loom/locks/issue-<N>/` to compute the "in-use" claim set.
- **Migration**: invocation unchanged. The "in-use" detection is now
  spawn-loop-aware; if you run `loom-clean` while the spawn loop is
  active, it will correctly refuse to remove worktrees for issues with
  live tasks.

### Retired (no port)

- `loom-validate-state` — validates a file that no longer exists. Removed.
  If you scripted against its exit code, replace with a presence check on
  `.loom/spawn-loop-state.json`.

## State-file consumers

If you have downstream code (not in the Loom repo) that reads
`.loom/daemon-state.json` or `.loom/progress/`, those reads will return
"file not found" after v1.0.0. The full per-consumer disposition is in
[`docs/migration/daemon-state-consumers.md`](daemon-state-consumers.md). The
short summary:

- **`.loom/daemon-state.json`** — replaced by `.loom/spawn-loop-state.json`
  (smaller, focused on running task list). Schema:
  `{started_at, running:[{issue, pid, started_at, token}]}`. There is **no
  pipeline_state, no warnings, no shepherds, no completed_issues**.
- **`.loom/progress/shepherd-*.json`** — replaced by
  `.loom/sweep-checkpoint/issue-<N>.json`. Schema is checkpoint-focused
  (last completed phase, recovery hints), not milestone-focused.
- **No compatibility shim is provided.** Re-creating a daemon-state shape
  from spawn-loop-state was considered and explicitly rejected (see the
  rationale in `daemon-state-consumers.md` §"Conclusion: what Phase 3
  deletes vs preserves"). The dropped fields described state that no
  longer exists; emitting a fake shape would mislead callers.

## Three migration paths

If you have a downstream copy of Loom that has not migrated yet, you have
three valid choices:

**(a) Migrate now (recommended for active deployments).** Switch your
automation to `/loom:sweep` + spawn loop + the GitHub Actions workflows
before your next `install-loom.sh` run against v1.0.0+. This is the
maintained path.

**(b) Defer migration via a pinned Loom version.** Stop running
`install-loom.sh` against your repo, or pin `loom` to a specific 0.9.x tag
in your install scripts. This opts you out of upgrades (no security or
correctness backports for 0.9.x), but it is a valid choice for short-lived
deployments.

**(c) Defer migration via coordination.** Open a comment on the Phase 4
tracker (#3382) and coordinate a deferral window with the Loom maintainer.
Soft-deprecation warnings will continue rendering in your logs during the
deferral; suppress with `LOOM_SUPPRESS_DEPRECATION=1` if needed. The
`/shepherd` markdown skill warning is intentionally non-suppressible — see
the suppression note below.

## Suppression of soft-deprecation warnings (0.9.x → v1.0.0 window)

If you are reading this guide before v1.0.0 has actually shipped, the
deprecated entry points still function but emit warnings. To silence:

```bash
export LOOM_SUPPRESS_DEPRECATION=1
```

This silences the Python (`loom_tools.common.deprecation.warn_deprecated`)
and shell (`.loom/scripts/lib/deprecation.sh`) warning emitters. The
`/shepherd` slash-command warning rendered by the markdown skill itself is
**not** suppressible by design — operators should explicitly migrate, not
silence.
