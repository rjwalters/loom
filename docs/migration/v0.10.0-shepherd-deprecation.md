## TL;DR

**The shepherd surface is removed.** The Python shepherd brain
(`loom_tools/shepherd/`), the `loom-shepherd` CLI, and the `/shepherd`
Claude Code slash command have all been **removed in v0.10.0**. They were
soft-deprecated in 0.9.x (#3376, Phase 2b). Their replacement is:

- **`/loom:sweep <issue>`** â€” single-issue lifecycle (Curator â†’ Builder â†’
  Judge â†’ Doctor â†’ Merge). Same phases the shepherd ran, maintained.

**Daemon mode is preserved, the Python brain is refactored out.** Loom needs
to be able to drive lifecycle work through *both* execution surfaces:

1. **Subagent dispatch** inside a single Claude Code session â€” `/loom:sweep`
   calls `Agent(subagent_type="loom-builder")` etc. Fast, single OAuth
   token (the parent's), single context.
2. **Daemon-managed tmux sessions** that launch Loom agents as **separate**
   Claude Code sessions â€” `./.loom/scripts/daemon.sh start` opens a tmux
   session whose panes each run a fresh `claude -p "/loom:..."` invocation,
   one rotated OAuth token per pane via
   [`./.loom/scripts/spawn-claude.sh`](../../.loom/scripts/spawn-claude.sh).

Both surfaces are first-class. The Python brain (`loom_tools/daemon_v2/`)
that historically coordinated the tmux session is removed; `daemon.sh`
becomes a thin shell-level launcher around the spawn loop + GitHub Actions
cron + token-rotated tmux panes. The user-facing API (`daemon.sh
{start,stop,status}`) is preserved. See
[How daemon mode works in v0.10.0](#how-daemon-mode-works-in-v0100).

**Why both surfaces?** Subagent dispatch is the right answer for batched
issue work where one operator session can chew through a queue. But Claude
Code subagents inherit the parent's `CLAUDE_CODE_OAUTH_TOKEN`, so a single
session is bound by one account's rate limits â€” fatal for multi-day runs.
The tmux-launched-separate-sessions surface gives each long-running worker
its own process, and `spawn-claude.sh` rotates the OAuth token at that
process boundary. **Token rotation only works at process spawn**, not within
a subagent dispatch tree â€” that constraint is the architectural reason
daemon mode survives.

If you were running `/shepherd <issue>` interactively, the equivalent is:

```text
/loom:sweep <issue>
```

If you were running `claude -p "/shepherd 123"` from a script, the
equivalent is:

```bash
claude -p "/loom:sweep 123" --dangerously-skip-permissions
```

If you were running `./.loom/scripts/daemon.sh start`, the API is unchanged
â€” same `start` / `stop` / `status` commands. The implementation is
different (see [How daemon mode works in v0.10.0](#how-daemon-mode-works-in-v0100));
no operator-side action required.

If you have downstream code that imports from `loom_tools.daemon_v2` or
`loom_tools.shepherd`, or that reads `.loom/daemon-state.json` or
`.loom/progress/`, those imports will break â€” see
[Per-CLI breaking changes](#per-cli-breaking-changes) and
[State-file consumers](#state-file-consumers).

## How daemon mode works in v0.10.0

The Rust `loom-daemon` binary runs as a long-lived background process,
managed by the operator's service manager of choice (systemd, launchd,
foreman, or a background shell). It does not poll the forge for ready
issues â€” operators (or `/loom:sweep` Stage -1 delegation) dispatch sweeps
explicitly via MCP.

**Process model:**

```
init/launchd â†’ loom-daemon  â”€â”€MCPâ”€â”€â†’  Claude Code session
                  â”‚                    (using /loom or /loom:sweep)
                  â”śâ”€â”€ SweepRegistry (in-memory BTreeMap)
                  â”śâ”€â”€ EventBus (tokio broadcast, 6 frozen topics)
                  â””â”€â”€ ReaperTask (30s tick â€” emits .exited / .crashed)
                  â”‚
                  â–Ľ
        On dispatch_sweep:
        fork+exec /loom:sweep N via spawn-claude.sh
        (token rotation at this process-spawn boundary)
```

**Dispatch flow:**

1. Operator (or `/loom:sweep` Stage -1) calls `mcp__loom__dispatch_sweep --issue N`.
2. The daemon picks a token from `.loom/tokens/.ranking` (or allowlist, or random) and fork+execs `claude -p "/loom:sweep N"` via `spawn-claude.sh`.
3. The child PID is registered in the in-memory `SweepRegistry`; a `sweep.global.dispatch` event is emitted.
4. The sweep child runs the lifecycle (Curator â†’ Builder â†’ Judge â†’ Doctor â†’ Merge), publishing `sweep.issue.{N}.phase` events as it advances.
5. On clean exit, the reaper (or the child itself) emits `sweep.issue.{N}.exited`; on non-zero exit, `sweep.issue.{N}.crashed`.
6. Operator (or any MCP client) can observe via `list_sweeps`, `get_sweep_status`, `subscribe_to_events`, `tail_sweep_log`, `tail_event_bus`.

**Support roles** (Champion, Curator, Judge, Auditor, Guide) run on cron via GitHub Actions workflows under `.github/workflows/loom-*.yml`. They are **not** dispatched by the daemon â€” the daemon is purely a sweep-dispatch + monitoring surface.

**Architectural framing**:

| Surface | Process model | Token model | Best for |
|---------|---------------|-------------|----------|
| Subagent dispatch (`/loom:sweep` in one session) | Single process, multiple subagents | Single OAuth token (parent's) | Operator-driven batches, â‰¤ hours of runtime, single-account operators |
| Daemon dispatch (`mcp__loom__dispatch_sweep` against `loom-daemon`) | Multiple processes (one per sweep child) | Rotated per process via `spawn-claude.sh` | Multi-day autonomous operation, multi-account rotation |

Both surfaces are first-class. Choose by runtime expectations:
- For an operator running a few-hour batch from their workstation with a single Claude account: `/loom:sweep` is faster, simpler, no daemon setup needed.
- For a multi-day autonomous run rotating across multiple Claude Pro/Max accounts: `mcp__loom__dispatch_sweep` against `loom-daemon` is the answer. `/loom:sweep` Stage -1 will auto-detect the daemon + pool and delegate dispatch transparently.

The architectural reason both exist: Claude Code subagents inherit the
parent's `CLAUDE_CODE_OAUTH_TOKEN`, so multi-account rotation only happens
at process-spawn boundaries. The subagent surface optimizes for fast
iteration within one token's budget; the daemon surface optimizes for
spreading load across many tokens over long time horizons.

## How to enable the replacements

### Daemon mode (multi-account autonomous dispatch)

The Rust `loom-daemon` binary is the supported multi-account orchestrator
for v0.10.0+. Start it from a terminal outside Claude Code via your
service manager of choice (systemd unit, launchd plist, foreman, or just
a background shell):

```bash
# In a background shell, or under your service manager:
loom-daemon
```

The daemon binds a Unix socket and serves IPC over it until stopped.

From a Claude Code session, drive it via MCP tools:

```
# Dispatch a sweep for one issue
mcp__loom__dispatch_sweep --issue 123

# List currently-dispatched sweeps
mcp__loom__list_sweeps

# Inspect a single sweep
mcp__loom__get_sweep_status --sweep_id <id>

# Subscribe to lifecycle events
mcp__loom__subscribe_to_events --topic "sweep.issue.*"

# Cancel a stuck sweep
mcp__loom__cancel_sweep --sweep_id <id>

# Tail a sweep's log
mcp__loom__tail_sweep_log --issue 123 --lines 200
```

The `/loom` skill (`defaults/.claude/commands/loom/loom.md`) wraps these
into an observer/dispatcher loop that polls the forge, dispatches ready
issues, and surfaces blockers + crashes to the operator.

The daemon **does not** generate work (no Architect / Hermit triggers),
**does not** maintain a `shepherd-N` pool, and **does not** drive support
roles. Each `mcp__loom__dispatch_sweep` request runs its own self-contained
`claude -p "/loom:sweep N"` child with its own rotated OAuth token. If you
need work generation, see the Phase 2d follow-up (#3381). Periodic support
roles run via the GitHub Actions cron workflows under
`.github/workflows/loom-*.yml`.

For the full wire protocol, IPC request/response variants, registry and
reaper semantics, and event taxonomy, see [`.loom/docs/daemon-reference.md`](../../.loom/docs/daemon-reference.md).


Checkpoints from #3373 are preserved across the rename â€” if a sweep child
crashes mid-flight and a `.loom/sweep-checkpoint/issue-<N>.json` exists, the
next run of the sweep skill resumes from the recorded phase.

## Per-CLI breaking changes

The nine operator CLIs that historically consumed `.loom/daemon-state.json`
or `.loom/progress/` were ported as part of Phase 3.1.x. Each has its own
PR and follow-up issue. Below is the field-level summary â€” for the
authoritative pre/post shape comparison, read the CLI's `--help` and the
linked PR.

### `loom-status` (#3399)

- **Before**: read `.loom/daemon-state.json` for shepherd state, pipeline
  state, warnings; cross-referenced with forge queries.
- **After**: reads `.loom/spawn-loop-state.json` for running tasks (just
  `{issue, pid, started_at, token}`), forge queries for everything else.
- **Dropped fields**: `shepherds.shepherd-N.*` (pool slots gone),
  `pipeline_state.*` (forge is source of truth), `warnings[]`,
  `total_prs_merged`, `last_architect_trigger`, `last_hermit_trigger`.
- **Renamed fields**: `shepherds[].issue` â†’ `tasks[].issue`,
  `shepherds[].task_id` â†’ `tasks[].pid`.
- **Migration**: re-script anything that consumed those dropped fields
  against `gh issue list --json` and `gh pr list --json` directly.

### `loom-backlog` (#3400)

- **Before**: read `.loom/daemon-state.json` for shepherd warnings about
  blocked PRs; cross-referenced with `.loom/issue-failures.json` for
  durable failure history.
- **After**: reads `.loom/issue-failures.json` only. The `daemon-state.json`
  cross-reference path is gone.
- **Dropped fields**: any `warnings[].context.requires_role` heuristics.
- **Migration**: most callers were already preferring `issue-failures.json`;
  this PR removes a fallback path that few callers exercised.

### `loom-stuck-detection` (#3401)

- **Before**: read `.loom/daemon-state.json` shepherd heartbeat fields and
  `.loom/progress/shepherd-*.json` milestone files.
- **After**: reads `.loom/spawn-loop-state.json` task pids (probing via
  `kill -0 <pid>`) and `.loom/sweep-checkpoint/issue-<N>.json` timestamps.
- **Dropped indicators**: `missing_milestone:worktree_created`,
  `extended_work` (no longer tractable without `last_phase` in state).
- **Retained indicator**: `stale_heartbeat` (now derived from checkpoint
  mtime, not progress-file mtime).
- **Migration**: thresholds are env-configurable as before; if you
  scripted against the JSON output, the `stale_heartbeat` block schema is
  unchanged, but the `extended_work` block is gone.

### `loom-completions` (#3402)

- **Before**: read shepherd `output_file` paths from daemon-state.json to
  produce shell completion data.
- **After**: reads `.loom/logs/sweep-issue-*.log` directly.
- **Breaking**: completion shape unchanged. If you wrote a completion
  cache that keyed on `output_file`, re-key on the log path.

### `loom-agent-metrics` (#3404)

- **Before**: daemon-state.json fallback for "is agent currently working".
- **After**: `activity.db` only. The fallback is gone.
- **Migration**: ensure `activity.db` is being populated by your spawn-loop
  / sweep runs (this is unchanged â€” sweep children write activity entries
  the same way shepherd children did).

### `loom-orphan-recovery` (#3414)

- **Before**: cross-referenced daemon-state.json's `shepherds[]` against
  the forge label set to find issues stuck in `loom:building` with no
  active shepherd.
- **After**: cross-references `.loom/spawn-loop-state.json` tasks against
  the forge.
- **Migration**: invocation unchanged (`--recover`, `--dry-run`, `--json`).
  Output schema unchanged.

### `loom-cleanup` (#3412, renamed from `loom-daemon-cleanup`)

- **Renamed**: `loom-daemon-cleanup` â†’ `loom-cleanup`. The old name is gone.
- **Before**: cleaned up `.loom/progress/shepherd-*.json` files, rotated
  `daemon-state.json` archives, emitted `session-rotation` events.
- **After**: cleans up `.loom/logs/sweep-issue-*.log` (configurable
  retention), purges stale `.loom/locks/issue-<N>/` dirs, no session
  rotation.
- **Migration**: update any cron entry or hook that referenced
  `loom-daemon-cleanup` by name. The new name is `loom-cleanup`.

### `loom-health-monitor` (#3413)

- **Before**: computed a composite health score from daemon-state.json,
  `.loom/health-metrics.json`, `.loom/alerts.json`.
- **After**: reads forge state only (queue depth, PR backlog) and writes a
  simplified `.loom/health-metrics.json` (smaller schema).
- **Dropped fields**: `stuck_agents`, `resource_availability` (these
  required state that no longer exists).
- **Migration**: re-tune your alerting thresholds. The new health score is
  not directly comparable to the old one.

### `loom-clean` (#3415)

- **Before**: read `daemon-state.json` to decide whether a worktree was in
  use by an active shepherd.
- **After**: reads `.loom/spawn-loop-state.json` (`tasks[].issue`) and
  `.loom/locks/issue-<N>/` to compute the "in-use" claim set.
- **Migration**: invocation unchanged. The "in-use" detection is now
  spawn-loop-aware; if you run `loom-clean` while the spawn loop is
  active, it will correctly refuse to remove worktrees for issues with
  live tasks.

### Retired (no port)

- `loom-validate-state` â€” validates a file that no longer exists. Removed.
  If you scripted against its exit code, replace with a presence check on
  `.loom/spawn-loop-state.json`.

## State-file consumers

If you have downstream code (not in the Loom repo) that reads
`.loom/daemon-state.json` or `.loom/progress/`, those reads will return
"file not found" after v0.10.0. The full per-consumer disposition is in
[`docs/migration/daemon-state-consumers.md`](daemon-state-consumers.md). The
short summary:

- **`.loom/daemon-state.json`** â€” replaced by `.loom/spawn-loop-state.json`
  (smaller, focused on running task list). Schema:
  `{started_at, running:[{issue, pid, started_at, token}]}`. There is **no
  pipeline_state, no warnings, no shepherds, no completed_issues**.
- **`.loom/progress/shepherd-*.json`** â€” replaced by
  `.loom/sweep-checkpoint/issue-<N>.json`. Schema is checkpoint-focused
  (last completed phase, recovery hints), not milestone-focused.
- **No compatibility shim is provided.** Re-creating a daemon-state shape
  from spawn-loop-state was considered and explicitly rejected (see the
  rationale in `daemon-state-consumers.md` Â§"Conclusion: what Phase 3
  deletes vs preserves"). The dropped fields described state that no
  longer exists; emitting a fake shape would mislead callers.

## Three migration paths

If you have a downstream copy of Loom that has not migrated yet, you have
three valid choices:

**(a) Migrate now (recommended for active deployments).** Switch your
automation to `/loom:sweep` + spawn loop + the GitHub Actions workflows
before your next `install-loom.sh` run against v0.10.0+. This is the
maintained path. If you had `./.loom/scripts/daemon.sh start` in your
automation, no operator-side change is needed â€” the API is preserved; only
the internals changed.

**(b) Defer migration via a pinned Loom version.** Stop running
`install-loom.sh` against your repo, or pin `loom` to a specific 0.9.x tag
in your install scripts. This opts you out of upgrades (no security or
correctness backports for 0.9.x), but it is a valid choice for short-lived
deployments.

**(c) Defer migration via coordination.** Open a comment on the Phase 4
tracker (#3382) and coordinate a deferral window with the Loom maintainer.
Soft-deprecation warnings will continue rendering in your logs during the
deferral; suppress with `LOOM_SUPPRESS_DEPRECATION=1` if needed. The
`/shepherd` markdown skill warning is intentionally non-suppressible â€” see
the suppression note below.

## Suppression of soft-deprecation warnings (0.9.x â†’ v0.10.0 window)

If you are reading this guide before v0.10.0 has actually shipped, the
deprecated entry points still function but emit warnings. To silence:

```bash
export LOOM_SUPPRESS_DEPRECATION=1
```

This silenced the Python (`loom_tools.common.deprecation.warn_deprecated`)
and shell (`.loom/scripts/lib/deprecation.sh` / `defaults/scripts/lib/deprecation.sh`)
warning emitters. Those helper modules were themselves **removed in v0.11.0**
along with the deprecated entry points they annotated, so this suppression
switch no longer applies. The `/shepherd` slash-command warning rendered by
the markdown skill itself was **not** suppressible by design â€” operators
should explicitly migrate, not silence.

Note that the 0.9.x deprecation warnings on `loom-daemon` and
`./.loom/scripts/daemon.sh` were emitted under the original (now revised)
plan that anticipated removing daemon mode entirely. In v0.10.0, those
warnings are **withdrawn** for `./.loom/scripts/daemon.sh` (the shell-level
surface is preserved). The Python `loom-daemon` CLI is genuinely removed,
so its warning escalates to an actual "command not found" error.

## `/loom:release` extension-point seams audit (#3573)

The retirement of the `/loom:release` skill (#3563, in favor of the
general-purpose `/repo:release` from `rjwalters/repo`) deleted
`defaults/.claude/commands/loom/release.md` and its example topic file
`defaults/hooks/example-context/topics/release.md`. Both carried five
`<!-- LOOM-EXTENSION-POINT: <name> -->` HTML-comment markers (added by #3506
/ proposal #3503): `pre-changelog-style`, `pre-push`, `post-push`,
`pre-github-release`, `post-summary`. This audit records whether anything
depended on them before they can be re-litigated.

**Finding â€” were the release seams used loom-internally? No.** The seams were
a *prose-composition convention*, not a runtime mechanism. Nothing in Loom
parsed the markers. The actual delivery vehicle is the generic
`defaults/hooks/methodology-inject.sh` `UserPromptSubmit` hook, which injects
any `.loom/context/topics/<name>.md` file whose name/pattern matches the
prompt â€” it has no knowledge of release phases or seam names, and #3563 left
it **untouched and intact**. The only two files that ever referenced the
seams were `release.md` and its own example topic, and #3563 removed them
together. Grepping `origin/main` after the merge confirms the sole surviving
mention of `LOOM-EXTENSION-POINT` is the historical `CHANGELOG.md` entry
(intentionally preserved as history); no code, script, or other doc reads the
markers. So no loom-internal machinery lost a dependency.

**Consumer guidance (the "if used" path).** A downstream consumer who authored
a `.loom/context/topics/release.md` targeting one of the named seams still has
the generic injection hook â€” their topic file continues to be injected on
matching prompts. What they lose is the Loom-shipped skill that defined the
named seam boundaries to compose against. Such consumers should move
release-time customization into their own repo's release command
(`/repo:release` from `rjwalters/repo`) or their own `CLAUDE.md`, rather than
relying on a Loom-defined seam. **Non-goal:** Loom will not re-add `release.md`
or a runtime injection hook to `/repo:release` â€” the repo is general-by-design
(see #3563 non-goals).

Provenance: seam origin #3503 / #3506; retirement #3563; this audit #3573.
